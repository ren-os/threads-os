# conversion.md — 転換率改善システム

LINE上の各フェーズの転換率を管理・改善するためのOS。
「友達追加→フォーム回答→相談→ASP成約」の各ステップの詰まりを発見・解消する。

---

## 改善対象の転換ポイント

```
LINE友達追加
    ↓ [転換率A] フォーム回答率
Googleフォーム回答
    ↓ [転換率B] 相談予約率
個別相談
    ↓ [転換率C] エージェント登録率
転職エージェント登録
    ↓ [転換率D] ASP成約率
ASP報酬確定
```

---

## 転換率の目標値（フェーズ別）

| 転換ポイント | 現状 | フェーズ1目標 | フェーズ2目標 |
|-----------|------|------------|------------|
| A. フォーム回答率 | TODO | 30%以上 | 50%以上 |
| B. 相談予約率 | TODO | 50%以上 | 70%以上 |
| C. エージェント登録率 | TODO | 60%以上 | 80%以上 |
| D. ASP成約率 | TODO | 40%以上 | 60%以上 |

> 全体ファネル目標（フェーズ1）：友達追加100人→ASP成約6件以上
> （100 × 30% × 50% × 60% × 40% ≒ 3.6 → 端数補正で6件目標）

## 転換率A：フォーム回答率の改善

### よくある離脱原因と対策

| 離脱原因 | 対策 |
|---------|------|
| フォームが長い | 質問を5つ以内に絞る |
| 何のために答えるかわからない | greeting.md のフォーム誘導文を改善する |
| フォームに気づかない | リッチメニューの①ボタンを最目立つ位置に配置 |
| 「まだ迷ってる」で止まる | 「迷ってる段階OK」の文言を強化する |

### フォーム回答率が20%未満の場合のアクション

```
① greeting.md のメッセージ3を見直す（フォームへの誘導文）
② フォームの質問数を削減する
③ 「フォームなしでメッセージで話してもらってもOK」の選択肢を増やす
④ step_messages.md のDay3でフォームへの再誘導を強化する
```

---

## 転換率B：相談予約率の改善

### よくある離脱原因と対策

| 離脱原因 | 対策 |
|---------|------|
| 相談の形式が怖い | consultation_flow.md の概要を事前に送る |
| 日程調整が面倒 | 候補日を多め（5つ以上）に出す |
| 「まだ準備できていない」という感覚 | 「準備不要」「話すだけでOK」を強調する |
| フォーム回答後の返信が遅い | フォーム回答後24時間以内に必ず返信する |

### 相談予約率が30%未満の場合のアクション

```
① フォーム回答後の返信スピードを確認する
② reply_templates.md「フォーム回答後の最初の返信」を見直す
③ 相談内容の説明（step_messages.md Day6）のタイミングを早める
④ 「断ってもOK・話すだけでもOK」の安心感をより強く伝える
```

---

## 転換率C：エージェント登録率の改善

### よくある離脱原因と対策

| 離脱原因 | 対策 |
|---------|------|
| エージェントへの不信感 | 相談内でエージェントの仕組みを丁寧に説明する |
| 「まだ転職を決めていない」 | 「情報収集だけでも登録OKなエージェントが多い」を伝える |
| 複数社の選定に迷う | 1社に絞って紹介する（選択肢を増やさない） |
| 紹介後のフォローが途切れる | 登録後2日以内に進捗確認メッセージを送る |

### エージェント登録率が40%未満の場合のアクション

```
① 相談内でのエージェント紹介トークを見直す（consultation_flow.md参照）
② knowledge/asp.md のエージェント選定フローを更新する
③ 登録後のフォロー頻度を上げる（3日→2日に短縮）
```

---

## 転換率D：ASP成約率の改善

### よくある離脱原因と対策

| 離脱原因 | 対策 |
|---------|------|
| エージェント登録後に連絡が途絶える | LINEで週1回の進捗フォローを続ける |
| 面接で落ちて諦める | フォローメッセージで継続を促す |
| 内定が出ても承諾しない | 相談に乗りながら判断をサポートする |
| 転職を中断する | 長期フォローに切り替える（broadcast.md参照） |

### ASP成約率が30%未満の場合のアクション

```
① エージェントへの紹介から入社まで、れんのフォロー頻度を確認する
② 「落ちた直後」の声かけを強化する（諦め防止）
③ knowledge/asp.md の成果発生タイミングを確認し
　 紹介するエージェントを見直す
```

## チャーン（ブロック・無反応）防止

### ブロックが増えるパターン

| 原因 | 改善方法 |
|------|---------|
| 配信頻度が多い | broadcast.md の配信回数を週1回に減らす |
| 内容が売り込み色が強い | TYPE A（ノウハウ）の比率を上げる |
| 返信に対する反応が遅い | 当日または翌日中に返信する |
| 有益な情報がない | broadcast.md の内容の質を上げる |

### ブロック率が5%を超えた場合のアクション

```
① broadcast.md の配信内容・頻度を見直す
② step_messages.md のDay1〜7の内容を短縮・精査する
③ greeting.md の挨拶文を見直す（第一印象が悪いと早期ブロック）
```

---

## 無反応ユーザーへの再アプローチ

### 判断基準

| 状況 | アクション |
|------|---------|
| 3日返信なし | reply_templates.md「3日返信なし」を送る |
| 1週間フォーム未回答 | reply_templates.md「1週間フォーム未回答」を送る |
| 7日間完全無反応 | broadcast.md の定期配信に切り替え |
| 1ヶ月無反応 | LINE配信を月1回以下に減らす（ブロック防止） |

---

## 月次改善サイクル

1. kpi.md の各転換率を確認する
2. 最も転換率が低いポイントを特定する
3. 上記の改善アクションから1〜2つ選んで実施する
4. 翌月に再計測して効果を検証する
5. improvement.md に結果を記録する
