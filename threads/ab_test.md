# ab_test.md — A/Bテスト管理

改善を自動ループで回すためのA/Bテスト管理ファイル。
`generate_posts.md` と `improvement.md` が毎回参照する。

---

## A/Bテストの基本設計

### テストの原則

- 1テストにつき1変数のみ変更する（複数変えると要因が特定できない）
- 最低10投稿分（各5本ずつ）のデータを集めてから判定する
- 勝ちパターンは「勝ちパターン保存」に移動し、次回から標準採用する
- テスト中は2案を交互に使う

### 判定基準

| 指標 | 勝ちの条件 |
|------|---------|
| フックA/B | 閲覧数の高い方（差が20%以上で確定） |
| CTAのA/B | LINE登録数に貢献した方（差が1件以上で優先） |
| 投稿時間A/B | プロフィール遷移率の高い方 |
| 投稿形式A/B | 保存率の高い方 |
| 画像・サムネA/B | プロフィール遷移率の高い方 |

## 次回テストキュー（優先順）

現在のテスト完了後に実施するテストを管理する。

| 優先度 | テスト変数 | 仮説 | 開始条件 |
|--------|---------|------|--------|
| 1 | フック型 | 数字 > 共感（閲覧数） | ABT-001完了後 |
| 2 | CTA文言 | ハードル最低 > 相談明示（登録率） | LINE開設後 |
| 3 | 投稿時間 | 夜 > 朝（プロフ遷移率） | 20本分データ後 |
| 4 | 投稿形式 | チェックリスト > ストーリー（保存率） | 各型10本後 |
| 5 | 画像あり/なし | 画像 > テキスト（プロフ遷移率） | 画像投稿開始後 |

## generate_posts.md が参照するルール（要約）

```
## A/Bテスト中の生成ルール

1. 「テスト状態：進行中」のテストがある場合、
   各回の投稿にA案とB案を交互に含める。

2. 「勝ちパターン保存」に記録されたパターンは
   次回から標準として優先採用する。

3. 「負けパターン記録」に記録されたパターンは
   使用しない。
```

---

## improvement.md が参照するルール（要約）

```
## A/Bテスト結果の改善反映ルール

1. 週次レビュー時に進行中テストのデータを集計する。
2. 勝者が確定したら「勝ちパターン保存」に移動し、
   improvement.md の「次回反映事項」に追記する。
3. 完了したテストは「改善履歴」に記録する。
4. 次のテストを「次回テストキュー」から選んで開始する。
```
